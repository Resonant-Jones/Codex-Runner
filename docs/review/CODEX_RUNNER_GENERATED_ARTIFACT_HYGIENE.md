# Codex Runner Generated Artifact Hygiene

## Status

- **Status:** generated-artifact hygiene note
- **Scope:** docs + ignore policy only
- **Runtime impact:** none
- **Home repository:** `Resonant-Jones/Campaign-Runner` (`/Volumes/Dev_SSD/Codex-Runner`)

This note labels the ashtray. It does not throw anything away. It does not give the scanner hands.

---

## 1. Purpose

Before any future session-log or runtime-artifact writer is added (e.g. a Guardian session-log stub), make explicit which paths in the working tree are **source** and which are **generated output**. This keeps generated evidence out of commits without weakening any source surface, and without widening any authority.

The governing rule:

> **Runtime outputs are evidence artifacts, not source authority.**

### `.pi/` — generated runtime output

`.pi/` is the Pi Loop Manager's run-output directory. Every `codexrun loop --dry-run` (or `--execute`) writes a new `.pi/runs/<run_id>/` folder. Its contents are receipts and evidence artifacts for that single run.

- `.pi/` is **generated output**, not source.
- `.pi/` should **not** be committed. It is now listed in `.gitignore`.
- Existing `.pi/runs/*` files are **not deleted** by this hygiene task. They remain on disk as local evidence until Chris decides otherwise.

### `.promptnomicon/` — authored scaffold (do not ignore)

`.promptnomicon/` holds three authored markdown files:

```txt
.promptnomicon/promptnomicon-steward.md
.promptnomicon/promptnomicon-steward-session.md
.promptnomicon/project-reality-footer.md
```

`README.md` states: "This repo now includes Promptnomicon Steward scaffolding under `.promptnomicon/`" and instructs operators to start a stewardship pass with `.promptnomicon/promptnomicon-steward-session.md`. That is an authored source surface, not generated output.

- `.promptnomicon/` is **not generated**. It is authored scaffold.
- `.promptnomicon/` is **not added to `.gitignore`**. Ignoring it would risk hiding authored material the README references.
- It is currently untracked. The recommended disposition is to **commit it as source** in a future commit grouping (it pairs naturally with the `README.md` change that advertises it), or for Chris to confirm it should stay developer-local. No action is forced here.

## 4. Rules for Future Session-Log / Runtime Writers

Any future runtime or session-artifact writer (e.g. a Guardian session-log stub) must follow these rules:

1. **Write only to designated generated-output locations.** Pi Loop writes under `.pi/runs/`. A future Guardian session log should write under an equivalingly clearly-named generated path (e.g. `.guardian/sessions/`), never into `src/`, `docs/`, `tests/`, or any spine source path.
2. **Generated output is evidence, not source.** It must not be committed unless a future task explicitly asks for a fixture or evidence sample.
3. **Add the generated path to `.gitignore` in the same slice that introduces the writer.** Do not ship a writer whose output silently appears as untracked noise.
4. **Never delete generated output from within a hygiene task.** Cleanup, if any, is a separate explicit decision by Chris.
5. **Never treat runtime output as architecture authority.** A receipt describes a run; it does not redefine contracts, schemas, or boundaries.

---

## 5. What Should Never Be Committed Unless Explicitly Intended

```txt
.pi/runs/*                          Pi Loop dry-run evidence
.guardian/sessions/* (future)       Guardian session logs, if/when added
any receipt.json / gate-receipts.json emitted at runtime
any validation.log, handoff.md, plan.md emitted at runtime
any copied task.yaml placed into a run dir
editor/OS files (.DS_Store, *.swp)  (already ignored)
```

The only exception is a **curated fixture or evidence sample** that a future task explicitly designates for commit (e.g. the existing `tests/fixtures/*.json` snapshots, which are authored and frozen, not generated).
